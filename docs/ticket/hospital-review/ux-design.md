# 全体フロー設計

## サービス全体の行動変容

```
【Before】
動画制作者: 動画完成 → スプシ共有 → メールで依頼
病院担当者: スプシ確認 → メールで返信
動画制作者: メール確認 → 修正 → 再度メール...

【After】
動画制作者: 動画完成 → 公開設定（ワンクリック）→ URL共有
病院担当者: URL開く → 確認 → OK/修正希望（システム上で完結）
動画制作者: 通知/一覧で確認 → 対応
```

## 3つのフェーズ

| フェーズ   | 主体       | 行動                             | 感情目標             |
| ---------- | ---------- | -------------------------------- | -------------------- |
| ① 公開     | 動画制作者 | 制作物を病院に公開する           | 「簡単に共有できた」 |
| ② レビュー | 病院担当者 | 確認してフィードバックする       | 「迷わず完了できた」 |
| ③ 対応     | 動画制作者 | フィードバックを確認して修正する | 「何をすべきか明確」 |

---

## フェーズ① 公開（動画制作者）

### ユーザー

**動画制作者（Contrea）**

- 制作物が完成した
- 病院担当者に確認してもらいたい
- 素早く共有して次の作業に移りたい

### 欲求と現状

| 欲求                           | 現状                                    | 課題         |
| ------------------------------ | --------------------------------------- | ------------ |
| 簡単に共有したい               | スプシをコピー、URLを作成、メールで送信 | 手順が多い   |
| どの病院に公開したか管理したい | メール履歴を探す                        | 把握しづらい |

### 行動設計

```
利用前: 制作物が完了ステータスになる
    ↓
利用中: 制作物一覧 → 三点メニュー → 公開設定 → 病院選択 → 保存
    ↓
利用後: URLを病院担当者に送付
```

### 方向性

> **「公開設定を、迷わず1分で完了できるシンプルな操作にする」**

### 感情設計

| タイミング         | 感じてほしいこと     |
| ------------------ | -------------------- |
| 公開設定を開いた時 | 「病院を選ぶだけか」 |
| 保存した時         | 「すぐ終わった」     |
| URLをコピーした時  | 「これを送ればOK」   |

## フェーズ③ 対応（動画制作者）

### ユーザー

**動画制作者（Contrea）**

- 病院からフィードバックが来たことを知りたい
- 何を修正すべきか把握したい
- 対応が完了したら再度確認してもらいたい

### 欲求と現状

| 欲求                       | 現状                           | 課題                 |
| -------------------------- | ------------------------------ | -------------------- |
| フィードバックを把握したい | メールを待つ、スプシを見に行く | 受動的、気づきにくい |
| 修正点を一覧で見たい       | スプシをスクロール             | 見落としやすい       |
| 対応完了を伝えたい         | メールで連絡                   | 手間                 |

### 行動設計

```
利用前: （将来）通知で「フィードバックが届きました」
    ↓
利用中: 制作物一覧 → 詳細画面 → フィードバック確認
        → 修正が必要なステップを把握 → 修正作業
    ↓
利用後: 修正完了 → 再公開（ステータスが「レビュー待」に戻る）
```

### 方向性

> **「フィードバック確認画面を、対応すべきことが一目でわかるチェックリストにする」**

### 感情設計

| タイミング                 | 感じてほしいこと             |
| -------------------------- | ---------------------------- |
| フィードバック一覧を見た時 | 「どこを直せばいいかわかる」 |
| ステップごとの指摘を見た時 | 「具体的で対応しやすい」     |
| 修正して再公開した時       | 「あとは確認を待つだけ」     |

# 病院担当者のUX設計

## 1. ユーザー定義

### プライマリユーザー

**病院の担当者**（事務長、入院管理担当など）

### 利用状況

| 項目       | 内容                                                               |
| ---------- | ------------------------------------------------------------------ |
| きっかけ   | 動画制作会社（Contrea）から「確認してください」とURLが送られてくる |
| タイミング | 業務の合間に確認する（専任ではない）                               |
| 環境       | PC / スマホ、ITリテラシーは様々                                    |
| 体制       | 複数人で分担して確認することもある                                 |

---

## 2. 欲求と現状

| 欲求                         | 現状の対処                      | 課題                 |
| ---------------------------- | ------------------------------- | -------------------- |
| 内容が正確か確認したい       | スプシを見て1行ずつチェック     | 面倒、見落としやすい |
| 修正点を伝えたい             | スプシに書き込む / メールで連絡 | 情報が分散する       |
| 確認が終わったことを伝えたい | 「確認しました」とメール        | 手動で連絡が必要     |
| 全体として問題ないか伝えたい | **明確な手段がない**            | ⚠️ 重要な課題        |

### 核心的な課題

> 「全体としてOK」「全体として修正が必要」を伝える方法がない

---

## 3. 行動設計

### 利用前

```
URLを受け取る → 時間を見つける → URLをクリック
```

**課題**:

- いつまでに確認すべきか不明
- 後回しにされがち

### 利用中

```
名前入力 → 制作物を選ぶ → ステップを1つずつ確認
→ 気になる点があればフィードバック入力
→ 最後まで見る → 「終わった」を伝えたい
```

**課題**:

- 最後に「全体としてどうか」を伝える場所がない
- 「見終わった」ことを明示的に伝えられない

### 利用後

```
「確認完了」をContreaに伝えたい → メール？電話？
```

**課題**:

- 完了報告の手段がバラバラ
- 双方で進捗が把握しにくい

---

## 4. 方向性

> **「この画面を、安心して確認完了を伝えられる場所にする」**

### 実現すること

- 「見たよ」だけでなく「このままでOK」「修正してほしい」を明確に伝えられる
- 確認が完了したことが記録に残り、双方に伝わる
- 複数人で確認しても、誰がどう判断したかわかる

---

## 5. 感情設計

| タイミング               | 感じてほしいこと                 | 設計方針                             |
| ------------------------ | -------------------------------- | ------------------------------------ |
| URLを開いた時            | 「シンプルでわかりやすい」       | 余計な情報を排除、病院名を明示       |
| ステップを見ている時     | 「ちゃんと確認できている」       | 画像・テキストを見やすく配置         |
| フィードバックを書いた時 | 「伝わる感じがする」             | 入力欄を目立たせ、即時保存           |
| **最後に確定した時**     | **「ちゃんと完了できた、安心」** | 確定アクションを明確に、完了感を演出 |

---

## 6. ステータス設計

### 制作物単位のステータス

| ステータス | 意味                     | 色       | トリガー                 |
| ---------- | ------------------------ | -------- | ------------------------ |
| レビュー待 | まだ確定していない       | グレー   | 初期状態                 |
| 修正希望   | 確認完了、修正してほしい | オレンジ | 「修正希望」ボタン押下   |
| OK         | 確認完了、問題なし       | グリーン | 「このままOK」ボタン押下 |

### なぜ制作物単位か

- ステップ単位だと判断が細かすぎる
- 「全体としてどうか」を伝えるのが本質的な欲求
- Contrea側も「この制作物は進めてOKか」を知りたい

## 8. 期待される効果

| 観点               | Before                           | After                        |
| ------------------ | -------------------------------- | ---------------------------- |
| 病院担当者         | 確認完了をメールで伝える         | システム上で明確に伝えられる |

# 制作物一覧画面 UX設計（病院担当者向け）

### ユーザーの状況

- 業務の合間に「動画の確認お願いします」とURLが来た
- 複数の制作物がある場合もある
- 「どれを見ればいいの？」「全部見ないといけないの？」
- 時間がないので**優先度が高いものから片付けたい**

### 核心的な欲求

> **「自分がやるべきことをすぐに把握して、さっさと終わらせたい」**

### 方向性

> **「この画面を、今やるべきことが一目でわかるタスクボードにする」**

### 設計方針

| 方針               | 実装                                                     |
| ------------------ | -------------------------------------------------------- |
| 未完了を目立たせる | 「対応が必要」セクションを上部に配置、青いボーダーで強調 |
| 件数を明示         | 「対応が必要（1件）」「対応済み（2件）」                 |
| 完了感を演出       | 全て完了時に「🎉 対応が必要な制作物はありません」を表示  |
| 対応済みは控えめに | opacity を下げて視覚的な優先度を下げる                   |

### 感情設計

| タイミング         | 感じてほしいこと         |
| ------------------ | ------------------------ |
| 一覧を開いた瞬間   | 「あ、これだけか」       |
| 未対応を見つけた時 | 「これをやればいいんだ」 |
| 対応を完了した時   | 「終わった！」           |
| 全て完了した時     | 「全部終わった、安心」   |
